# Guess The Flag

Guess The Flag - учебный веб-проект для тренировки распознавания флагов стран. Приложение состоит из backend API на Go, frontend на React/TypeScript и базы данных PostgreSQL.

## Постановка задачи

Требовалось разработать веб-приложение, в котором пользователь может зарегистрироваться, начать игровую сессию, получить вопрос с изображением флага, выбрать или ввести ответ и увидеть результат. Дополнительно проект должен хранить данные об игроках, странах, вопросах и ответах, поддерживать статистику прохождения и быть подготовленным к локальному запуску через конфигурационные файлы и Docker.

Основные функциональные требования:

1. Регистрация и аутентификация пользователя.
2. Создание игровой сессии.
3. Получение вопросов по флагам стран.
4. Проверка ответа пользователя.
5. Поддержка двух режимов ответа: ввод с клавиатуры и выбор из вариантов.
6. Сохранение результатов игры и обновление статистики.
7. Разделение клиентской и серверной частей проекта.

## Выбранная методология

Для проекта наиболее подходит итеративно-инкрементальная Agile-методология с элементами прототипирования.

Выбор этой методологии обоснован тем, что веб-игра состоит из нескольких независимых, но связанных подсистем: интерфейса, API, базы данных, авторизации, игровой логики, проверки ответов и деплоя. Такие части удобно разрабатывать короткими итерациями: сначала получить минимально работающий прототип игры, затем постепенно добавлять регистрацию, хранение данных, разные режимы ответа, тесты и документацию.

Методология позволяет быстро проверять работоспособность проекта на каждом этапе. Для учебного проекта это особенно важно, потому что ошибки в архитектуре, API или модели данных можно обнаружить до того, как будет написан большой объем кода.

## Этапы реализации и примененные методы

### 1. Анализ предметной области

На первом этапе была выделена основная пользовательская задача: игрок должен увидеть флаг и определить страну. Также были определены сущности предметной области: пользователь, игра, вопрос, ответ, страна, изображение флага и таблица результатов.

Примененные методы:

- анализ требований;
- декомпозиция предметной области;
- моделирование сущностей;
- формализация пользовательских сценариев.

### 2. Проектирование архитектуры

Проект был разделен на backend, frontend и слой хранения данных. Backend реализует API и бизнес-логику, frontend отвечает за пользовательский интерфейс, PostgreSQL хранит пользователей, страны, вопросы, ответы и статистику.

Примененные методы:

- архитектурная декомпозиция;
- проектирование REST API;
- проектирование схемы базы данных;
- разделение ответственности между слоями приложения.

### 3. Проектирование базы данных

Для хранения данных использована PostgreSQL. В проекте есть SQL-миграции, которые описывают структуру таблиц и позволяют воспроизводимо поднимать схему базы данных.

Примененные методы:

- реляционное моделирование данных;
- нормализация сущностей;
- миграционный подход к управлению схемой;
- выделение репозиториев для доступа к данным.

### 4. Реализация backend

Серверная часть написана на Go. Она включает маршрутизацию HTTP-запросов, middleware, сервисы пользователя и игры, работу с JWT, хеширование паролей, репозитории и игровую логику.

Примененные методы:

- модульное программирование;
- слоистая архитектура;
- обработка ошибок;
- использование JWT для аутентификации;
- хеширование паролей с помощью bcrypt;
- алгоритмическая проверка текстовых ответов, включая расстояние Дамерау-Левенштейна.

### 5. Реализация frontend

Клиентская часть создана на React и TypeScript. Интерфейс содержит экраны авторизации, начала игры, прохождения вопросов, просмотра результатов и отладки флагов.

Примененные методы:

- компонентный подход;
- типизация данных API;
- управление состоянием на стороне клиента;
- маршрутизация экранов;
- визуальное прототипирование пользовательского сценария.

### 6. Тестирование и отладка

В проекте присутствуют backend-тесты для игровой логики, вариантов игры, состояния игровой сессии и работы с репозиториями. Для проверки API также подготовлены Swagger-описание и Postman-коллекция.

Примененные методы:

- модульное тестирование;
- интеграционная проверка API;
- тестирование граничных случаев;
- ручная проверка пользовательского сценария;
- отладочные эндпоинты для визуальной проверки SVG-флагов.

### 7. Подготовка к запуску и деплою

Для проекта добавлены конфигурационные файлы, Dockerfile, docker-compose и отдельные настройки для production-среды. Это упрощает воспроизведение окружения и запуск приложения.

Примененные методы:

- контейнеризация;
- конфигурационное управление;
- разделение dev- и prod-настроек;
- документирование процесса запуска.

## Возникшие сложности и способы их решения

1. Необходимо было связать frontend, backend и базу данных в единый пользовательский сценарий. Это решалось через явное проектирование API-контрактов и типизацию ответов на стороне клиента.
2. Проверка текстовых ответов требовала учитывать опечатки. Для этого был применен алгоритмический метод сравнения строк на основе расстояния Дамерау-Левенштейна.
3. Данные о странах и флагах нужно было хранить так, чтобы их можно было быстро получать во время игры. Для этого используется загрузка данных в базу и in-memory-хранилище на стороне сервиса игры.
4. Игровая сессия должна исключать повторение уже использованных стран. Эта проблема решена через хранение состояния сессии и проверку использованных country_id.
5. Проекту требовалась воспроизводимость запуска. Для этого добавлены миграции, конфигурационные файлы и Docker-окружение.

## Архитектура проекта

```text
Guess-The-Flag/
├── backend/                 # Go API
│   ├── cmd/server/          # точка входа сервера
│   ├── internal/api/        # HTTP handlers, router, middleware
│   ├── internal/service/    # бизнес-логика пользователя и игры
│   ├── internal/db/         # модели, запросы и репозитории
│   ├── internal/config/     # конфигурация
│   └── migrations/          # SQL-миграции
├── frontend/                # React/TypeScript приложение
│   ├── src/components/      # UI-компоненты
│   ├── src/services/        # API-клиент
│   └── src/types/           # типы данных
├── deploy/                  # материалы для деплоя
├── docker-compose.yml       # локальный запуск
├── docker-compose.prod.yml  # production-запуск
└── config.yml               # конфигурация приложения
```

## Быстрый запуск

### Требования

- Go 1.24 или выше.
- Node.js 14 или выше.
- PostgreSQL 12 или выше.
- Git.

### Backend

```bash
cd backend
go mod download
go run cmd/server/main.go --config=../config.yml
```

Backend по умолчанию доступен на `http://localhost:8080`.

### Frontend

```bash
cd frontend
npm install
npm start
```

Frontend по умолчанию доступен на `http://localhost:3000`.

### Тестирование

```bash
cd backend
go test ./...
```

```bash
cd frontend
npm test
```

## Вывод

Итеративно-инкрементальная Agile-методология хорошо подходит для Guess The Flag, потому что проект развивается через последовательное добавление работающих возможностей. Сначала можно реализовать минимальный игровой цикл, затем расширять его авторизацией, базой данных, режимами игры, статистикой, тестами и деплоем. В результате каждый этап дает проверяемый результат, а примененные методы соответствуют задачам конкретного этапа разработки.
